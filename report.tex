\begin{abstract}
A three-dimensional DEM solver is implemented and parallelised using OpenMP.
The solver uses a spring-dashpot contact model and semi-implicit Euler
integration. The code is tested using simple cases and shows correct behaviour.
Parallelisation improves performance for larger systems, though speedup is limited
for small sizes due to overhead.
\end{abstract}

\section{Introduction}

Granular systems are common in many real-world applications such as powders,
sand, and industrial processes. Understanding their behaviour is important
in areas like civil engineering, material science, and physics.

The Discrete Element Method (DEM) is widely used to simulate such systems by
tracking individual particles and computing forces acting on them. At each
timestep, interactions between particles are calculated and their motion is updated.

A major issue with DEM is that computing all pairwise interactions scales as
$\mathcal{O}(N^2)$. This becomes very expensive as the number of particles increases.
Because of this, simulations with large numbers of particles can take a long time.

To improve performance, parallel computing techniques can be used. In this work,
OpenMP is applied to parallelise different parts of the solver. The aim is to
reduce runtime while maintaining correct physical behaviour.

\section{Mathematical Model}

Each particle follows Newton's second law:

\begin{equation}
m_i \frac{d\mathbf{v}_i}{dt} = \mathbf{F}_i
\end{equation}

The total force includes gravity and contact forces from other particles.

\subsection{Contact Model}

When two particles come into contact, they overlap slightly. The overlap is given by:

\begin{equation}
\delta = R_i + R_j - d_{ij}
\end{equation}

If $\delta > 0$, a force is applied using a spring-dashpot model:

\begin{equation}
F_n = k_n \delta - \gamma_n v_n
\end{equation}

The spring term models elastic behaviour, while the dashpot term accounts
for energy loss during collisions. This helps simulate realistic bouncing behaviour.

\section{Numerical Method}

A semi-implicit Euler scheme is used to update the system:

\begin{align}
\mathbf{v}_i^{n+1} &= \mathbf{v}_i^n + \frac{\mathbf{F}_i}{m_i}\Delta t \\
\mathbf{x}_i^{n+1} &= \mathbf{x}_i^n + \mathbf{v}_i^{n+1}\Delta t
\end{align}

This method is simple to implement and reasonably stable for small timesteps.

Each timestep consists of:
\begin{itemize}
\item Resetting forces
\item Adding gravity
\item Computing particle interactions
\item Handling wall collisions
\item Updating velocities and positions
\end{itemize}

The timestep is chosen small enough to maintain stability and accuracy.

\section{Profiling}

Profiling was carried out to identify which parts of the code take the most time.
It was observed that the particle interaction loop dominates the runtime.

This is expected since the number of interactions increases rapidly with the
number of particles. Other parts of the code take much less time in comparison.

\section{Parallelisation}

OpenMP is used to parallelise loops in the code. Loops where each particle
is independent are straightforward to parallelise.

However, the particle interaction loop is more complicated because multiple
threads may try to update the same particle simultaneously. This leads to
race conditions.

To avoid this, atomic operations are used. While this ensures correctness,
it introduces some overhead and reduces performance gains.

\section{Conclusion}

The DEM solver produces correct results for different test cases such as free fall,
constant velocity, and bouncing motion.

Parallelisation using OpenMP improves performance, especially for larger systems.
However, the speedup is limited due to atomic operations and thread overhead.

Further improvements could include reducing the number of interaction checks or
using better data structures to improve efficiency.
